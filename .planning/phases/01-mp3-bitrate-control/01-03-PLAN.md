**Edit A — Add props-seeding $effect in the `<script>` block**

Find the existing block of $effects around lines 59–65 that reads from `d` (app.props?.default). After the last closing brace of those effects (but still inside the `<script>` tag), add a new $effect that seeds app.mp3Bitrate from props when it has not been set yet:

```typescript
// Seed bitrate from server default on first load (or whenever props arrive).
// app.mp3Bitrate === 0 means "not yet set"; overwrite with server's configured default.
// Falls back to 128 if props are unavailable (e.g., server offline on first render).
$effect(() => {
    if (app.props && app.mp3Bitrate === 0) {
        const serverDefault = Number(app.props.cli.mp3_bitrate) || 128;
        app.mp3Bitrate = serverDefault;
    }
});
```

Place this immediately after the existing `$effect(() => { if (!d) return; ... })` block (the one that seeds lm_batch_size, synth_batch_size, peak_clip) so all props-seeding effects are grouped together.

**Edit B — Insert bitrate <select> into the format row template**

Find the format row in the template (lines ~862–891). It currently ends with the format `<select>` and then `</div>`. Insert the bitrate control BEFORE the format `<select>` (so the order reads: batch → peak clip → bitrate → format → end):

```svelte
{#if app.format === 'mp3'}
<select
    bind:value={app.mp3Bitrate}
    title="MP3 encoding bitrate. Higher = better quality, larger file. 128 kbps is the server default. Changes apply to this request only."
>
    {#if ![96,128,160,192,256,320].includes(app.mp3Bitrate)}
    <option value={app.mp3Bitrate}>{app.mp3Bitrate} kbps (server default)</option>
    {/if}
    <option value={96}>96 kbps</option>
    <option value={128}>128 kbps</option>
    <option value={160}>160 kbps</option>
    <option value={192}>192 kbps</option>
    <option value={256}>256 kbps</option>
    <option value={320}>320 kbps</option>
</select>
{/if}
```

The conditional `{#if}` block at the top of the option list renders an extra option only when `app.mp3Bitrate` is not one of the six standard presets. This ensures that any valid server-configured bitrate (e.g. 64, 80, 112) is visible and selectable; when `app.mp3Bitrate` is a standard preset value the extra option is suppressed and the selector shows the usual six choices.

Insert this block immediately before the `<select bind:value={app.format} ...>` element in the format row. Do not modify the format select itself.

Exact insertion point: the line immediately before `<select` that has `bind:value={app.format}`.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | props-seeding: Number(app.props.cli.mp3_bitrate) coercion | mitigate | `Number(...) || 128` fallback. If props.cli.mp3_bitrate is a non-numeric string or missing, defaults to 128 (server's own default). The server validates the final submitted value independently (Plan 01). |
| T-03-02 | Spoofing | Bitrate select: <select> with fixed <option> values | accept | Options are hard-coded preset values plus one conditional dynamic option derived from the server's own props response; no free-text input. Server validates anyway as defense-in-depth. |
| T-03-03 | Tampering | app.mp3Bitrate bound to select — could be mutated via browser console | accept | Local inference tool, no auth model. Server-side validation (Plan 01) is the authoritative guard. |
</threat_model>
